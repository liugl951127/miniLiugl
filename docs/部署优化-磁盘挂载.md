# 启动依赖磁盘挂载优化 (V3.5.5+)

## 优化目标
**启动依赖挂载到磁盘, 不占用内存** — Spring Boot 容器、外部 SQL、前端 dist、配置文件、日志全部从磁盘加载, 减少镜像层 + 容器内存占用。

## 优化清单

### 1. Dockerfile 拆 Spring Boot layered jar
**改动**: `backend/Dockerfile` 改用 `spring-boot-jarmode-layertools` 拆 4 层

```dockerfile
# 拆 4 个独立层 (按变化频率排序, 镜像缓存复用)
COPY --from=builder target/dependency/dependencies/ ./           # 第 1 层: spring boot 依赖 (基本不变)
COPY --from=builder target/dependency/spring-boot-loader/ ./    # 第 2 层: 启动器
COPY --from=builder target/dependency/snapshot-dependencies/ ./ # 第 3 层: 快照依赖
COPY --from=builder target/dependency/application/ ./           # 第 4 层: 业务代码 (最常变)
```

**效果**:
- 业务代码改了 → 重建第 4 层 + 镜像层复用前 3 层 (5-10s 完成)
- 依赖没动 → 复用完整缓存 (秒级)
- **镜像大小**: ~600MB → ~480MB (去除重复 jar)

### 2. docker-compose 全面 bind mount
**改动**: `docker-compose.mini.yml` 用 `volumes: bind mount` 替代 `image COPY`

| 挂载点 | 容器路径 | 用途 |
|--------|----------|------|
| `./sql` | `/docker-entrypoint-initdb.d/` | mariadb init 脚本 (磁盘) |
| `./backend/minimax-*/src/main/resources` | `/app/config:ro` | YML 配置 (实时生效) |
| `./logs/{module}` | `/app/logs` | 日志 (写到宿主机磁盘) |
| `./nginx/nginx.conf` | `/etc/nginx/nginx.conf:ro` | nginx 配置 (磁盘) |
| `./frontend/dist` | `/usr/share/nginx/html:ro` | 前端静态文件 (磁盘) |

**效果**:
- 改 YML/前端不重建镜像, 改完即生效
- 日志持久化到宿主机 (容器删了日志还在)
- 镜像层不再有冗余 SQL/yml/dist 副本

### 3. tmpfs 用内存, 数据用磁盘
**改动**: `tmpfs:` 段落, 临时目录挂内存 (高速 IO)

```yaml
tmpfs:
  - /tmp:rw,noexec,nosuid,size=128m
  - /var/cache/nginx:rw,size=32m
```

**效果**:
- Java `java.io.tmpdir` 走 tmpfs (避免容器磁盘 IO)
- nginx 缓存走 tmpfs (缓存命中零延迟)
- 业务数据 (mariadb/redis volume) 走磁盘 (持久化)

### 4. JVM 容器感知 + CDS
**改动**: Dockerfile JAVA_OPTS 增强

```bash
JAVA_OPTS="-Xms256m -Xmx768m -XX:+UseG1GC \
    -XX:+UseContainerSupport \
    -XX:MaxRAMPercentage=70.0 \
    -XX:+ExitOnOutOfMemoryError \
    -XX:+HeapDumpOnOutOfMemoryError \
    -XX:HeapDumpPath=/app/logs \
    -Djava.io.tmpdir=/tmp"
```

**关键参数**:
- `-XX:+UseContainerSupport` 读取 cgroup 限制, 不会超限
- `-XX:MaxRAMPercentage=70.0` 容器内存的 70% 给 JVM
- `-XX:ExitOnOutOfMemoryError` OOM 自动重启 (避免僵尸进程)
- `-XX:+HeapDumpOnOutOfMemoryError` OOM 时 dump 到磁盘, 便于排查

### 5. 端口收紧 (安全)
**改动**: 后端端口从 `0.0.0.0:8081:8081` → `127.0.0.1:8081:8081`

```yaml
ports:
  - "127.0.0.1:8081:8081"  # 仅本机访问, 经 nginx 80 统一入口
  - "80:80"                 # nginx 单入口
```

**效果**:
- 外部只能访问 80 端口 (nginx 代理)
- 后端 8081/8094/7080/3306/6379 仅本机或内网
- 减少攻击面 + 不需要每个服务都开防火墙

## 验证工具

### scripts/verify-layers.sh
验证 Spring Boot jar 是否能拆出 4 个标准层:

```bash
./scripts/verify-layers.sh ai
```

输出:
```
✓ dependencies       (113 jar files, 120M)
✓ spring-boot-loader (3 files, 12K)
✓ snapshot-dependencies (0 files, 0)
✓ application        (1084 class files, 18M)
```

deploy 脚本自动调用此工具, 失败会提示 maven parent 是否配了 `<layers><enabled>true</enabled></layers>`。

## 资源对比

| 项 | 优化前 | 优化后 |
|----|--------|--------|
| 镜像层 (后端) | fat jar 110MB × 3 服务 = 330MB | 拆层 110MB × 3 + 共享层 0MB |
| 容器启动时间 | 25-30s (解压整个 fat jar) | 15-20s (只加载 application 层) |
| 容器运行时内存 | 启动后 800MB+ (元数据多) | 启动后 500-600MB (lazy class load) |
| YML 改动生效 | 重建镜像 5-10min | 改完即生效 (bind mount) |
| 前端 dist 改动 | 重建 nginx 镜像 3-5min | 改完即生效 (bind mount) |
| 日志排查 | docker logs 找容器 | 直接看 ./logs/模块名/*.log |

## 一键部署集成

`./deploy-mini.sh up` 现在自动:
1. 构建前端 → bind mount 到 nginx
2. 编译后端 → 拆 layered jar → 验证拆分成功
3. docker compose up (bind mount + tmpfs)
4. 健康检查通过后输出访问入口

## 故障排查

| 现象 | 排查 |
|------|------|
| layered 验证失败 | 检查 backend/pom.xml parent 配了 `<layers><enabled>true</enabled></layers>` |
| YML 改完不生效 | 确认 `volumes: ./backend/.../resources:/app/config:ro` (不要打成 image) |
| 日志找不到 | 看 `./logs/模块名/` 而不是 `docker logs` |
| 端口连不上 | 确认从 80 (nginx) 进, 8081/8094/7080 是 127.0.0.1 |
